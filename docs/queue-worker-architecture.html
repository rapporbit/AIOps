Redis Streams 任务队列 · Worker 架构

代码: app/queue/redis_streams.py · app/diagnosis_worker.py · app/core/distributed_limiter.py

1 · 端到端通路

生产者 XADD 入队 → 按优先级分流 → 消费组多 Worker 消费 → 成功落库 / 失败进死信

生产者 / 结构容器 优先级子流 Worker 成功落库 死信

队列消息只是"触发器"，任务的事实状态机（pending → running → succeeded / failed、attempts 计数）落在 Postgres 的 diagnosis_tasks 表。XACK 把消息从消费组的 PEL（Pending Entries List）移除；Worker 崩溃时未 ACK 的消息会被下一轮 XAUTOCLAIM 重新认领。

2 · 单条消息的处理与容错

每个 Worker 对一条消息的完整生命周期：认领 → 校验 → 执行 → 落库 / 重试 / 死信

容错有三道闸：幂等（任务已 succeeded 直接 ACK 跳过，避免重复写 Evidence）、重试（失败且 attempts < max_attempts → 改回 pending、重新 XADD 入同优先级流、再 ACK 原消息）、死信（任务不存在 / 超过重试上限 / 最终失败 → 写 aiops:incident_tasks:dlq，保留 raw_item 供排查重放）。

3 · 三个关键概念

Worker 到底指什么

Worker 不是线程、不是协程，而是一个独立的操作系统进程——python -m app.diagnosis_worker 启动的 DiagnosisWorker 实例。它的本质是 Redis 消费组 diagnosis-workers 里的一个 consumer，用一个唯一的 consumer name（worker-1/worker-2/…，来自 --name 或环境变量 DIAGNOSIS_WORKER_ID）向 Redis 标识自己。

  • 进程内是单线程 asyncio 事件循环：靠 await 做 I/O 并发，没有多线程。一个 Worker 进程同一时刻只跑一个诊断主循环（count=1，一次取一条）。
  • 横向扩展靠多进程：起 3 个进程就是 3 个 worker，共享同一个消费组，Redis 自动把不同消息分发给不同 consumer，互不重复。
  • 每个 Worker 自带心跳：后台协程定期写 aiops:worker:diagnosis-workers:{name}:heartbeat（带 TTL），用于 /queue/status 判断存活——心跳只是运行态信号，不是事实库。

所以一个 worker = 一个进程 = 消费组里的一个 consumer，靠 consumer name 唯一标识、靠心跳证明活着、靠下面的分布式槽与同伴协调全局并发。

优先级抢占的两段式读

Redis Streams 的 XREADGROUP 原生不支持"跨多条流按优先级取"。BLOCK 模式监听多条流时，规则是"任一条有数据就返回"，不保证返回的是最高优先级那条。于是 read_tasks() 用两段式绕开这个限制：

  • 第一段（非阻塞优先级扫描）：按 critical → high → normal → low 顺序，对每条流各发一次 block=None 的非阻塞 XREADGROUP，命中即立刻返回，绝不去看更低优先级的流。这就是"严格抢占"——只要 critical 有货，永远先于 low 被处理。
  • 第二段（阻塞兜底）：四条流全空时，对所有流发一次 block=5000ms 的阻塞读，把空转交给 Redis 内核，零 CPU。一旦被唤醒，不直接用这次的结果，而是回到主循环重新走第一段——这样即使阻塞读唤醒时返回的是 low，下一轮扫描仍会先把这期间到达的 critical 取走。

两段配合：第一段保证优先级正确性，第二段保证空闲时不烧 CPU。代价是每次扫描多发几次轻量命令，但诊断任务本身要跑几十秒，这点开销可忽略。

分布式并发槽（distributed slot）

问题根源：asyncio.Semaphore 只在单进程内有效。一旦 uvicorn --workers 4 或起多个 Worker 进程，每个进程各有一份信号量，全局并发被放大 N 倍，限流形同虚设。解决办法是把"槽位"放到 Redis 上，让所有进程共享同一个上限。

  • 数据结构：每种资源一个 ZSET aiops:limiter:{resource}。member = 唯一 token（host:pid:uuid），score = 该槽的过期时间（毫秒）。占用数 = ZSET 基数。
  • 原子抢槽：用一段 Lua 脚本在 Redis 服务端原子完成「ZREMRANGEBYSCORE 清过期 → ZCARD 数当前 → 没满则 ZADD 占一个槽」，避免 check-then-act 竞态；用服务端 TIME 取时间，规避客户端时钟漂移。
  • TTL + 心跳续期：槽带 TTL，进程崩溃来不及释放时到期自动清掉，不会永久泄漏；长任务运行期间后台协程定期把自己 token 的 score 往后推（refresh_interval 默认 30s，TTL 默认 90s），防止被误清。正常结束在 finally 里 ZREM 释放。
  • 两种获取语义：wait=False 抢不到立刻抛 DistributedLimitBusy（用于同步入口，满了就拒绝/入队）；wait=True 抢不到就轮询等待（Worker 用这个，尊重全局上限、满了排队）。
  • pause / resume：通过 contextvar 暴露当前槽句柄，等待人工审批这类不确定时长的阻塞时可以先 pause() 让出槽（停心跳 + 释放），审批结束 resume() 重新抢回，避免空占并发。
  • 容错 fail-open：Redis 不可达时直接放行并告警。理由是整条链路本就强依赖 Redis，真挂了一切都不工作，限流层不该再叠一层硬失败把诊断也拖死——宁可短暂失去全局上限，也不让限流器成为新的单点。

Worker 执行诊断时正是用 async with distributed_slot("worker_diagnosis", limit=worker_diagnosis_concurrency, wait=True) 包住实际诊断调用：无论起多少个 Worker 进程，同时真正在跑的诊断数都不超过这个全局上限。
