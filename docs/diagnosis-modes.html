Fast / Deep 双诊断模式 · 可视化详解

Multi-Agent AIOps Platform V3 · L4 智能体层 · DiagnosisMode = {fast, deep}

fast 快速闭环 — 单 Skill 智能体的 Plan → Execute → Replan 循环,每步由 LLM 决策、可反思可重规划,适合日常诊断与通用排障。

deep 深度根因 — 多 Agent 并行取证 → 确定性归并打分 → LLM 根因裁决 → 处置建议的有向图,结构化证据链全程可追溯,适合生产重大故障的 RCA。

一、模式路由 · 一次请求如何分流

请求携带 diagnosis_mode。resolve_effective_mode() 受全局开关 deep_diagnosis_enabled 约束:deep 在开关开启时走独立 Deep 图,否则回落 fast 并打 group_agent_reserved 标记。

二、Fast 模式 · Plan-Execute-Replan 循环

单一 Skill 智能体的动态规划闭环。Skill Router 选方向 → Planner 出 4-6 步计划 → Executor 逐步取证 → Replanner 复盘,可继续 / 切 Skill / 收尾。带防死循环计数与 Harness 强制收敛。

Fast 节点逐一

节点职责关键机制
① Skill Router
LLM		 从告警 + 历史经验选中对应 Skill,确定本次诊断方向。 快速路径:简单问题直接产 response 跳过后续;查不到回退 generic_oncall。
② Planner
LLM		 从 Skill.playbook 提取种子,LLM 生成 4-6 步具体计划写入 state.plan。 LLM 失败用 planner_fallback_plan() 兜底,保证总能出计划。
③ Executor
LLM		 执行 plan[0],调用工具收集信息,结果 (step,result) 追加进 past_steps。 默认并行:只读工具按 ToolMeta.concurrency_safe 分批 asyncio.gather;可回落串行链路。
④ Replanner
LLM		 评估进度,输出 Act:continue / finish / reroute。 快速路径:剩余步数多且上步成功时跳过 LLM 直接继续(fast_path_threshold);超 agent_max_steps 强制收尾。

三、Deep 模式 · 多 Agent 并行取证有向图

串行前段建立上下文 → 规则路由派遣专业 Agent → fan-out 并行取证 → join 归并打分 → LLM 根因裁决 → 规则化处置 → 纯格式化报告。确定性优先,LLM 只在 RCA 一处兜决策。

Deep 节点逐一

节点职责关键机制
① Incident Manager
规则		 Worker 路径从 Postgres 查 task,充实 incident_group_id / incident_id;SSE 手动路径直接放行。 DB 异常不抛,仅记 transition 降级。
② Correlation Context
LLM		 聚合同组其它告警 + LLM Wiki 经验召回,告诉 RCA "这不是孤立事件"。 双保险 fallback,无依赖也能产 evidence。
③ Evidence Plan
规则		 关键词匹配决定派遣集合:cpu/mem→metric,log/error→log,docker/dns→infra+log,sop→runbook。 强信号("全面诊断")广播全部;无匹配默认 metric+log;_dispatch_guard 跳过未派遣 Agent。
四个专业 Agent
LLM · 并行		 Log / Metric / Infra / Runbook 四个隔离一次性 subagent,各自只产对应类型 Evidence。 operator.add reducer 让多 Agent 并发写 evidences 自动合并,互不干扰。
④ Evidence Reducer
规则		 每条 evidence → 候选,按类型基础分降序 (metric 1.0 > infra 0.90 > log 0.85 > runbook 0.75 > history 0.60),截 top-5。 error evidence 标记保留供识别失败,确定性无 LLM。
⑤ RCA Judge
LLM		 "SRE 根因判定法官",只看 evidence summary 不看原文,输出 root_cause / ranked / reasoning / confidence。 LLM 失败取 Reducer 最高分候选确定性兜底。
⑥ Remediation Planner
规则		 按技术关键词 (redis/mysql…) 匹配处置模板,每步标 [只读] 或 [写操作·需人工]。 硬约束:任何写操作 requires_human_confirm=True。
⑦ Report
无 LLM		 纯格式化:现象 / 根因 / 候选 / 证据链 (引用 [ev_i]) / 处置 / 元数据。 全链路可追溯,Agent 成功失败统计入报告。

四、双模式对比

维度fast · 快速闭环deep · 深度 RCA
图结构单 Agent · Plan-Execute-Replan 循环多 Agent · 串行→fan-out→join→串行 有向无环
入口节点Skill RouterIncident Manager
决策范式每步 LLM 动态决策 + 反思规则路由为主,LLM 仅 RCA 一处
取证编排逐步增量,单步内工具并行一批 fan-out,4 Agent 完全并行
证据处理串行累积进 past_stepsReducer 评分去重 → top-5 候选
根因判定Replanner 收尾时隐含生成专门 RCA Judge 节点(LLM + 确定性兜底)
处置建议报告中隐含专门 Remediation Planner + 人工确认闸门
报告生成LLM 动态生成Report 节点纯格式化(无 LLM)
故障恢复reroute 切 Skill 重规划各节点确定性降级兜底
LLM 调用量多(Router/Planner/Executor/Replanner 多轮)少(仅 Correlation + RCA Judge)
典型耗时较长 · 多轮迭代较短 · 一批并行
适用场景日常诊断 · 通用排障生产重大故障 · 需完整证据链追溯

参数默认作用

agent_max_reroutes1fast Skill 切换上限
